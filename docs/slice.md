## Итоговая оценка

Сейчас проект находится на стадии **рабочего игрового прототипа / vertical slice**.

Основной игровой цикл уже можно пройти полностью:

```text
Меню → загрузка кадра → выбор времени → проверка →
начисление очков → следующий раунд → итоговый экран
```

Но до MVP, описанного в [Project Research.md](C:/Kurage/CinemaGuess/docs/Project%20Research.md:85), проект пока не дошёл: отсутствуют база данных, импорт контента, filmstrip, внешнее хранилище, сохранение результатов и работа с правами на кадры.

## Что реализовано частично

### Каталог контента

Есть шесть фильмов и шесть кадров:

- [movies.ts](C:/Kurage/CinemaGuess/data/movies.ts:8)
- [frames.ts](C:/Kurage/CinemaGuess/data/frames.ts:8)

Этого достаточно для демонстрации механики, но не для нормальной игры. Поскольку игра тоже состоит из шести раундов, за одну сессию пользователь увидит весь каталог.

Кроме того, в `public/frames` присутствует неиспользуемый `1_drive.jpg`.

### Скрытие правильного ответа

Требование документа выполняется на базовом уровне: браузер сначала получает только публичные данные, а правильная позиция вычисляется сервером.

Однако полноценной серверной игровой сессии нет:

- сервер не создаёт `roundId`;
- не запоминает, какой кадр был выдан конкретному игроку;
- `/api/guess` принимает любой существующий `frameId`;
- идентификаторы вида `frame-1` предсказуемы;
- один ответ можно отправлять повторно.

Для локального прототипа этого достаточно. Для leaderboard или публичной версии — нет.

### Оптимизация изображений

`next/image` оптимизирует выдачу изображений браузеру, но исходные файлы подготовлены неоднородно: некоторые занимают примерно 1 МБ. Нет автоматической конвертации, ограничения размера и контроля разрешения.

### Работа с ошибками

Пользователь видит понятное сообщение, но техническая причина теряется:

```tsx
catch {
  setErrorMessage("Не удалось загрузить раунд.");
}
```

Логирования, мониторинга и диагностического идентификатора ошибки пока нет.

## Что отсутствует относительно плана

### Filmstrip

В документе предлагается кадр, filmstrip и ползунок ([строка 34](C:/Kurage/CinemaGuess/docs/Project%20Research.md:34)).

Сейчас присутствует только обычный HTML-ползунок. Предварительно сгенерированной киноленты или набора миниатюр нет.

### PostgreSQL / Supabase

Рекомендуемая схема:

```text
movies
frames
playable_rounds
guesses
users
leaderboard_entries
```

не реализована. Вместо базы используются два массива TypeScript.

Следствия:

- каталог нельзя обновлять без изменения кода;
- результаты не сохраняются;
- отсутствуют миграции и ограничения целостности;
- нет истории игр и статистики;
- нет leaderboard.

### Offline ingest pipeline

Рекомендованный процесс:

```text
видео → ffprobe/FFmpeg → PySceneDetect →
кандидаты кадров → ручная проверка → публикация
```

полностью отсутствует. В проекте нет Python-скриптов, FFmpeg-команд, импорта метаданных и механизма ручного одобрения кадров.

По плану даже первая версия должна иметь небольшой Python-импортёр ([строка 58](C:/Kurage/CinemaGuess/docs/Project%20Research.md:58)). Поэтому формально текущая версия ещё не закрывает весь объём «first playable prototype» из документа, хотя сама игровая механика уже playable.

### Cloudflare R2

Изображения находятся в `public/frames`. Для прототипа это проще и нормально, но рекомендуемая схема с объектным хранилищем не реализована.

### TMDb / Wikidata

Названия, годы и длительности введены вручную. Нет:

- внешних идентификаторов;
- автоматического импорта;
- проверки метаданных;
- TMDb attribution;
- информации об источнике записи.

### Права на контент

Это самый серьёзный разрыв с исследованием.

Документ рекомендует начинать с public-domain или явно лицензированных фильмов. Текущий каталог состоит из современных коммерческих фильмов, а в проекте нет сведений:

- откуда получены кадры;
- на каком основании они используются;
- разрешена ли публичная публикация;
- кто правообладатель;
- какие территории и способы использования разрешены.

Для локального учебного прототипа это не мешает разработке. Для публичного размещения — главный блокер.

### Тестирование

Автоматических тестов нет:

- нет unit-тестов формулы очков;
- нет тестов API;
- нет проверки связей `frame.movieId → movie.id`;
- нет end-to-end теста полного игрового цикла.

Сборка проходит, но это проверяет компиляцию, а не корректность поведения.
